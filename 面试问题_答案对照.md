## 模块一：项目架构与技术选型

### Q1：为什么后端选 Python FastAPI 而不是 Spring Boot 或 Go？

**简单回答：**
FastAPI 是 Python 里最快的 Web 框架之一，开发速度快，代码量少。它原生支持 async/await（异步处理），文件上传和接口响应的效率比 Django 高很多。而且它自动生成 Swagger 文档，别人 Fork 我的开源项目马上就能看到 API 接口说明。

**深入回答（加分项）：**
- FastAPI 基于 Starlette，性能接近 Node/Go，但开发效率是它们的 3-5 倍
- 内置 Pydantic 做数据校验，前端传错参数会自动报错
- 自动生成 Swagger/ReDoc 文档，开源项目必备

### Q2：为什么不直接用 Celery + Redis 做分布式任务队列？

**简单回答：**
因为项目初期不需要。BackgroundTasks 是 FastAPI 自带的轻量方案，适合中小规模任务。我现在用线程池处理转换任务，对于个人部署或者小团队完全够用。如果以后用户量大了，再平滑迁移到 Celery + Redis。

**瓶颈说明：**
- 单机线程池的缺点是：同时能跑的任务数有限（我限制了 2 个并发）
- 如果 10 个人同时上传大文件，后面的人需要排队等待
- 但免费方案本来就不支持大规模的并行转换，够用就行

**如果面试官问"用户量大了怎么办"：**
> 答：我做了线程池隔离和任务队列长度监控，如果并发数超过阈值，会返回 503 排队提示。未来可平滑迁移到 Celery + Redis 进行横向扩展（加机器）。

### Q3：前端为什么选 React + Ant Design 5，而不是 Vue + Element Plus？

**简单回答：**
React 生态最成熟，Ant Design 5 是业界最完善的企业级 UI 组件库，拖拽上传、进度条、卡片等开箱即用，开发效率最高。我做过对比，Ant Design 的主题定制能力（Design Tokens）比 Element Plus 灵活很多。

**关于 Next.js（SSR）：**
文档转换工具不需要 SEO 优化（用户不需要在百度上搜"在线转换工具"来找这个网站），所以不用 Next.js。React + Vite 的 SPA 方案加载快、交互流畅，更适合工具型网站。

### Q4：如果前端断网了，后端还在跑吗？转换完成后怎么通知前端？

**简单回答：**
- 后端不受影响，任务会继续在服务器上跑
- 断网恢复后，前端从 localStorage 读取任务 ID，重新轮询 `/status/{task_id}` 获取结果
- 如果转换已完成，直接展示下载链接

**关键代码思路：**
```
localStorage 存 task_id → 断网 → 恢复网络 → 轮询 /api/status/{id} → 结果
```

### Q5：如果后端 FastAPI 服务挂了，前端怎么优雅降级？

**简单回答：**
前端有一个健康检查机制：
1. 页面启动时访问 `/health` 接口
2. 如果失败，页面顶部显示"服务维护中"的横幅，禁用上传按钮
3. 每分钟重试一次，恢复后自动解除禁用

**代码位置：** `frontend/src/pages/Home/index.tsx` 的 useEffect 中实现了定时 health check。

### 你必须背下来的底层知识

**FastAPI 选型：**
FastAPI 基于 Starlette，原生支持 async/await，非阻塞 I/O 适合文件读写和 API 响应。比 Django（同步）更适合高并发文件上传和状态轮询。自动生成 Swagger 文档，对开源项目友好。

**BackgroundTasks vs Celery：**
- BackgroundTasks：轻量线程池，适合中小任务，简单够用
- Celery + Redis：分布式消息队列，适合任务量大、需持久化、多机扩展
- 我的项目现在用 BackgroundTasks，将来可平滑迁移到 Celery

**断网处理：**
如果前端断网，后端任务仍在跑。方案是用 localStorage 存储任务 ID，网络恢复时主动轮询 `/status/{task_id}`。这要在面试时主动说出来。

**健康检查：**
前端 useEeffect 定时访问 `/health`，失败时显示"服务维护中"并禁用上传。

## 模块二：高还原度文档转换引擎

### Q6：pdf2docx 怎么实现 PDF→Word 排版级转换的？

**简单回答（易懂版）：**
pdf2docx 的工作原理是这样的：它先读取 PDF 文件，然后分析每一页——这段话在什么位置、字体多大、这里有没有表格、那里有没有图片。最后把这些信息重新排版成 Word 格式。它不是截图，是真正的"识别+重建"。

**它怎么处理不同内容：**
- **段落文字**：按坐标和字体识别，保留字号、加粗、斜体
- **表格**：通过检测表格线框来识别，然后重建表格
- **图片**：直接从 PDF 中提取图片数据，放入 Word

**扫描件（纯图片PDF）：**
不行。如果 PDF 是扫描件（就是拍照/扫描的图片），pdf2docx 识别不了文字，因为图片里没有"文本层"。这时候会报错提示"PDF 为纯图片，请使用 OCR 服务"。

### Q7：PyMuPDF (fitz) 渲染 PDF→PPT，一页一张图还是保留文字？

**简单回答：**
一页一张图。我用 300DPI 高清渲染，每页 PDF 变成一张高清图片，然后插入到 PPT 的每张幻灯片中。

**优点：** 排版 100% 还原，PDF 长什么样 PPT 就长什么样
**缺点：** PPT 变成"图集"，文字不可编辑

**为什么这么做：**
因为 PDF→PPT 的核心需求是"演示"，不是"编辑"。保留原始排版比可编辑更重要。如果要可编辑，用户应该用 PDF→Word。

### Q8：LibreOffice 命令行桥接用什么命令？

**核心命令：**
```bash
soffice --headless --convert-to pdf --outdir /输出目录 /输入文件.pptx
```

**参数说明：**
- `--headless`：无 GUI 模式（服务器没有屏幕，必须加）
- `--convert-to pdf`：指定转换成 PDF
- `--outdir`：指定输出目录
- `--norestore`：禁用文档恢复（省内存，我加的优化）
- `--safe-mode`：安全模式，禁用扩展（省内存，我加的优化）

**LibreOffice vs Microsoft Office 的排版差异：**
这个处理不了，LibreOffice 和 Office 的排版引擎本身就有差异。我提供的方案是：如果用户追求 100% 的 Office 排版还原，建议在本地用 Office 打开。LibreOffice 转换保证的是"可用"而非"像素级一致"。

### Q9：python-pptx 和 python-docx 作为 PPT→Word 回退方案，能提取图片和表格吗？

**简单回答：**
不能。python-pptx 只能提取纯文本字符串，所有布局、图片、表格、颜色都会丢失。所以我把它叫"**回退方案**"，不是"完美替代方案"。

**回退时机：**
当 LibreOffice 不可用时（比如内存不足），才用这个方案保底。至少用户能拿到文字内容，而不是什么都拿不到。

### Q10：6种转换路径具体是哪些？

1. PDF → Word ✅（pdf2docx，排版级还原）
2. PDF → PPT ✅（PyMuPDF 高清渲染，图片级保留）
3. Word → PDF ✅（LibreOffice 命令行转换）
4. PPT → PDF ✅（LibreOffice 命令行转换）
5. PPT → Word ✅（python-pptx 提取文本）
6. PDF 拆分 ✅（pypdf 按页拆分）

### Q11：带密码的 PDF、损坏的 PDF 怎么处理？

**加密 PDF：** 在 `pdf_to_word.py` 的开头用 `pypdf.PdfReader` 检查 `reader.is_encrypted`，如果是加密的，直接抛异常："PDF 文件已加密，请先解密后再上传"。

**损坏的 PDF：** PyMuPDF 会抛出异常（如 `fitz.fitz.FileDataError`）。我在 try-except 中捕获所有异常，统一返回："文件格式损坏，请检查 PDF 是否正常"。

### Q12：pdf2docx 转换中文 PDF 时乱码怎么办？

**解决方案：** 在部署时安装中文字体。具体在 `nixpacks.toml` 中加了：
```bash
apt-get install -y fonts-wqy-zenhei fonts-liberation fonts-dejavu-core
```
然后执行 `fc-cache` 刷新字体缓存。

> 面试时主动说：**"我在转换前注入中文字体 wqy-zenhei，确保中文不变成方块。"** 这是面试官最想听到的工程细节。

## 模块三：高性能异步任务架构与大文件处理

### Q13：怎么实现 2GB+ 大文件流式写入？FastAPI UploadFile 默认存内存还是磁盘？

**简单回答：**
- FastAPI 的 `UploadFile` 默认把文件放在内存中，小文件没事，大文件会撑爆内存
- 我的方案：**分块读取，边读边写磁盘**

**具体实现：**
```python
# 每次只读 1MB，写入磁盘
with open(save_path, 'wb') as buffer:
    while chunk := await file.read(1024 * 1024):  # 每次 1MB
        buffer.write(chunk)
```

这样 2GB 的文件也只会占用 1MB 的内存。

**关键代码位置：** `backend/app/api/endpoints.py` 的 `/api/upload` 接口。

### Q14：TaskManager 为什么是线程安全的？

**简单回答：**
因为 FastAPI 的 BackgroundTasks 运行在线程池中，多个任务可能同时修改任务状态（比如同时更新进度）。我用 `threading.Lock`（线程锁）保护对任务字典的增删改操作，确保一次只有一个线程能修改数据。

**代码位置：** `backend/app/tasks/manager.py`
```python
with self._lock:  # 这行就是锁
    self._tasks[task_id] = task
```

### Q15：100 个用户同时轮询 /status，后端扛得住吗？

**简单回答：**
扛得住，因为：
1. `/status` 接口只是从内存字典读数据，不涉及磁盘 I/O，非常快
2. 前端实现了**指数退避**策略：轮询间隔从 1s → 2s → 4s → 8s 递增，不是每秒都请求
3. 数据量很小：每个任务只存 task_id、status、progress 几个字段

**前端代码思路：**
```javascript
// 轮询间隔逐渐增大
let delay = 1000;
polling();  // 1秒后
delay = 2000;  // 2秒后
delay = 4000;  // 4秒后
```

### Q16：转换过程中 LibreOffice 崩溃（OOM），TaskManager 怎么感知？

**简单回答：**
我用 `subprocess.run()` 的 `timeout` 参数设置了 300 秒超时。超时会抛 `TimeoutExpired` 异常。如果进程被系统杀掉（OOM），exit code 是 137，我检测到这个代码后立即把任务标记为"失败"并清理临时文件。

**关键代码：**
```python
result = subprocess.run(cmd, timeout=300)
if result.returncode == 137:  # OOM 被 kill
    task_manager.update_status(task_id, 'failed', '内存不足，转换失败')
```

### Q17：两种最难处理的异常情况

**情况一：文件格式伪装**
用户把 `.exe` 重命名为 `.pdf` 上传。我在两个环节检测：
1. **后端**：用 `python-magic` 或检查文件头魔数，不只依赖扩展名
2. **前端**：上传前检查扩展名白名单

**情况二：路径注入攻击**
文件名包含 `../../../etc/passwd`。防御方案：
- 不使用原始文件名存储，而是用 `task_id` + 扩展名作为存储文件名
- 代码：`get_safe_filename(task_id, original_filename)` 在 `file_utils.py` 中

> 面试中主动说出这两个情况，面试官会觉得你很有安全意识。

## 模块四：云原生部署与容器化适配

### Q18：Railway 的部署机制？Nixpacks 是什么？

**简单回答：**
Railway 是一个"自动化部署平台"，你把代码推到 GitHub，它自动拉取、构建、部署。它用 Nixpacks 自动检测项目类型（Python 还是 Node），不需要写 Dockerfile。

**Nixpacks vs Dockerfile：**
- Dockerfile：需要自己写，配置复杂
- Nixpacks：自动识别，一行配置都不用写就能部署
- 我选 Nixpacks 是因为省事，对于 Python 项目只需要指定根目录和启动命令

### Q19：LibreOffice OOM (exit=137) 怎么解决的？

**为什么发生：** LibreOffice 启动需要 150MB+ 内存，Railway 免费容器只有 256MB，超了就被 OOM Killer 杀掉。exit=137 就是"被系统杀掉"的信号（128 + 9 = 137）。

**解决方案三个层面：**

| 层面 | 方案 | 效果 |
|------|------|------|
| 参数 | `--norestore` + `--safe-mode` | 减少启动内存 30% |
| 架构 | TaskManager 限制并发数=2 | 防止多任务抢内存 |
| 工程 | apt-get 安装中文字体 | 防止中文渲染异常占用额外内存 |

### Q20：中文字体在 Railway 怎么安装？

在 `nixpacks.toml` 中配置：
```toml
[phases.install]
cmds = [
  "apt-get update -qq",
  "apt-get install -y -qq libreoffice-writer libreoffice-impress fonts-wqy-zenhei",
]
```
Railway 的构建阶段有 root 权限，可以用 `apt-get install` 安装系统软件包。

### Q21：Railway 睡眠策略怎么处理？

Railway 免费版如果 15 分钟无请求，服务会进入睡眠。解决方案：
1. **最佳方案**：升级付费计划，开启 Always On
2. **免费方案**：通过 GitHub Actions 每 5 分钟访问一次 `/health` 接口，保持服务活跃

### Q22：如果未来要迁移到 K8s，TaskManager 内存字典有什么问题？

**问题：** TaskManager 是单机内存存储，容器重启后所有任务状态丢失。在 K8s 多副本环境下，不同 Pod 之间的 TaskManager 互不相通。

**改造方案：** 把内存字典替换为 Redis。任务状态存到 Redis 里，所有 Pod 共享同一个 Redis。这在代码层面只需要改 TaskManager 的实现，接口完全不用动。

## 模块五：前端交互与工程化

### Q23：Design Tokens（设计令牌）怎么组织的？

**简单回答：**
在 `index.css` 的 `:root` 中定义所有设计变量，比如：
```css
:root {
  --color-primary: #4F46E5;
  --color-success: #10B981;
  --radius-md: 10px;
  --shadow-lg: 0 10px 15px ...;
}
```
所有组件都引用这些变量，而不是直接用具体的颜色值。

**如果做深色模式：** 只需要改：
```css
[data-theme="dark"] {
  --color-bg: #0F172A;
  --color-text: #FFFFFF;
}
```
所有组件自动适配，不需要改任何组件代码。

### Q24：拖拽上传的霓虹边框和粒子动效怎么实现的？

**纯 CSS + HTML 实现，不用 Canvas：**
- **霓虹边框**：`conic-gradient` 渐变 + `@keyframes borderRotate` 关键帧动画
- **粒子动效**：三个固定位置的 `div` 元素用 `@keyframes floatParticle` 动画来回飘动
- **性能开销**：几乎没有，纯 CSS 动画由 GPU 加速

**代码位置：** `frontend/src/components/FileUploader/index.tsx`

### Q25：进度轮询怎么做的？React.memo 有用吗？

**实现方式：** 每秒轮询 `/api/status/{task_id}`，更新进度条。我用 `React.memo` 包裹了 `ProgressBar` 组件，只有当 `progress` 值变化时才重新渲染。

**代码思路：**
```tsx
const ProgressBar = React.memo(({ progress, status }) => {
  return <Progress percent={progress} ... />;
});
```

### Q26：历史记录存在哪里？下载 URL 是 blob 还是服务器地址？

- **历史记录**：存在 `localStorage`，只存 task_id、文件名、状态、时间，**不存文件本身**
- **下载链接**：指向服务器地址 `/api/download/{task_id}`，不生成 blob URL，避免撑爆浏览器内存

### Q27：Ant Design Upload 二次发送会内存复制吗？

**优化方案：** 用 `beforeUpload` 拦截 Upload 组件的默认上传行为，返回 `false` 阻止自动上传，然后自己用 Axios 构造 FormData 发送，避免二次复制。

**关键代码：**
```tsx
beforeUpload: (file) => {
  handleFile(file);  // 直接处理文件
  return false;      // 阻止 Ant Design 自动上传
}
```

## 模块六：开源治理与 API 规范

### Q28：怎么保证 API 文档清晰？

FastAPI 自动生成 Swagger 文档，我通过 Pydantic 模型定义了每个接口的请求/响应格式：
```python
class ConvertRequest(BaseModel):
    task_id: str
    conversion_type: str  # 必须是 pdf_to_word / pdf_to_ppt 等

class TaskStatusResponse(BaseModel):
    task_id: str
    status: str          # processing / completed / failed
    progress: int        # 0-100
```

启动后端后访问 `/docs` 就能看到完整的交互式 API 文档。

### Q29：Git Workflow 怎么管理？

- `main` 分支：稳定版
- `formatconverter` 分支：当前开发分支
- 功能开发在 feature 分支，测试完成后合并到开发分支

**Issue 处理：** 如果有人提 Issue，我会先确认是不是 Bug，如果是就创建 fix 分支修复，合并后关闭 Issue。

### Q30：如果别人 Fork 项目，怎么确保能跑起来？

我在项目根目录提供了：
- `backend/requirements.txt` — Python 依赖清单
- `README.md` — 详细的环境配置和启动步骤
- Vite 配置了 API 代理，本地开发直接 `npm run dev` + `py -3 run.py` 就能启动

以后还会加 `docker-compose.yml`，一行命令跑起全栈。

### Q31：Release 怎么发布？

**步骤：**
1. 在 `package.json` 中更新版本号（v1.0.0）
